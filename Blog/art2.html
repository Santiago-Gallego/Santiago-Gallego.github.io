Optimización de Consultas en SQL Server

Escrito por Santiago Gallego

¿Por qué optimizar consultas?

En bases de datos con miles de registros, consultas mal escritas pueden ralentizar tu aplicación hasta en un 500%. SQL Server ofrece herramientas como el Execution Plan para diagnosticar problemas.

Técnicas Clave

1. Índices Estratégicos

Crea índices en columnas usadas frecuentemente en WHERE, JOIN y ORDER BY:

CREATE INDEX idx_usuario_email ON Usuarios(email);

2. Evita SELECT *

Selecciona solo las columnas necesarias:

-- Mal
SELECT * FROM Productos;

-- Bien
SELECT id, nombre, precio FROM Productos;

Análisis de Rendimiento

Usa estas consultas para monitorear:

-- Consultas más lentas
SELECT TOP 10 
  query_stats.query_hash,
  query_stats.execution_count,
  query_stats.total_elapsed_time / query_stats.execution_count AS avg_duration
FROM sys.dm_exec_query_stats AS query_stats
ORDER BY avg_duration DESC;

Índices Columnstore

Revolucionan el rendimiento en data warehouses (OLAP):

-- Creación de índice columnstore
CREATE CLUSTERED COLUMNSTORE INDEX CCI_Transacciones 
ON Transacciones;

-- Consulta optimizada para columnstore
SELECT 
    ProductoID,
    SUM(Cantidad) AS TotalVendido,
    AVG(PrecioUnitario) AS PrecioPromedio
FROM Transacciones
WHERE Fecha BETWEEN '2023-01-01' AND '2023-12-31'
GROUP BY ProductoID;

Parameter Sniffing Problem

Solución a un problema común en procedimientos almacenados:

-- Problema
CREATE PROCEDURE GetClientesPorRegion
    @RegionID INT
AS
BEGIN
    SELECT * FROM Clientes WHERE RegionID = @RegionID;
END;

-- Solución 1: Usar variable local
CREATE PROCEDURE GetClientesPorRegion_Opt
    @RegionID INT
AS
BEGIN
    DECLARE @LocalRegionID INT = @RegionID;
    SELECT * FROM Clientes WHERE RegionID = @LocalRegionID;
END;

-- Solución 2: OPTION (OPTIMIZE FOR UNKNOWN)
CREATE PROCEDURE GetClientesPorRegion_Opt2
    @RegionID INT
AS
BEGIN
    SELECT * FROM Clientes 
    WHERE RegionID = @RegionID
    OPTION (OPTIMIZE FOR UNKNOWN);
END;
